# 第二步：开始教学

## 第一讲：FastAPI简介与哲学：为什么是“Fast”API？

### 概念

**FastAPI**​ 是一个用于构建 API 的现代、快速（高性能）的 Web 框架，基于 Python 3.8+ 的标准类型提示（Type Hints）。

它的“Fast”体现在三个层面：

1. **开发速度**：极简的代码即可创建功能完整的 API。
    
2. **运行速度**：基于 Starlette（异步 Web 框架）和 Pydantic（数据验证库），性能与 Node.js 和 Go 的框架相当。
    
3. **学习速度**：设计直观，利用强大的编辑器自动补全和类型检查，减少错误。
    

### 原理

FastAPI 的“快”建立在两大核心技术之上：

1. **Python 类型提示**：这是其核心魔法。你只需用标准的Python语法声明一个变量的类型（如 `name: str`），FastAPI 就会利用这些信息来自动完成：
    
    - **请求解析**：从 HTTP 请求中提取数据并转换为正确的 Python 类型。
        
    - **数据验证**：检查数据是否符合声明的类型和约束，如果无效则自动返回清晰的错误。
        
    - **文档生成**：自动为你的 API 生成交互式文档。
        
    
2. **异步支持**：原生支持 `async`/`await`语法，可以轻松处理高并发 I/O 操作（如数据库查询、调用其他API），而不会阻塞整个应用，从而获得极高的性能。
    

### 例子

下面是一个最简单的FastAPI应用，它展示了其核心特性。

1. **安装**​ (这一步是准备工作，在代码之前完成)
    
    ```
    pip install fastapi
    pip install "uvicorn[standard]"
    ```
    
2. **最小应用代码**​ (保存为 `main.py`)
    
    ```
    from fastapi import FastAPI
    from pydantic import BaseModel
    
    # 1. 创建 FastAPI 应用实例
    app = FastAPI()
    
    # 2. 定义一个数据模型
    class Item(BaseModel):
        name: str
        price: float
        is_offer: bool = None  # 可选参数
    
    # 3. 定义路径操作装饰器
    @app.get("/")
    def read_root():
        return {"Hello": "World"}
    
    @app.get("/items/{item_id}")
    def read_item(item_id: int, q: str = None):  # 4. 使用类型提示定义路径参数和查询参数
        return {"item_id": item_id, "q": q}
    
    @app.put("/items/{item_id}")
    def update_item(item_id: int, item: Item):  # 5. 使用Pydantic模型定义请求体
        return {"item_name": item.name, "item_id": item_id}
    ```
    
3. **运行应用**
    
    ```
    uvicorn main:app --reload
    ```
    
    - `main`：你的 Python 文件 `main.py`
        
    - `app`：在文件中创建的 `FastAPI`实例对象
        
    - `--reload`：使服务器在代码更改后自动重启，仅用于开发。
        
    
4. **查看自动生成的交互式文档**
    
    启动服务后，打开浏览器访问：
    
    - Swagger UI 文档: `http://127.0.0.1:8000/docs`
        
    - ReDoc 文档: `http://127.0.0.1:8000/redoc`
        
    

### 总结

- **核心价值**：FastAPI 通过**Python类型提示**，实现了**声明式编程**。你只需“声明”你想要的数据格式，框架自动处理解析、验证、序列化和文档，极大地提升了开发效率和代码健壮性。
    
- **关键特点**：
    
    1. **高性能**：基于 Starlette 和 Pydantic，是现有 Python Web 框架中速度最快的之一。
        
    2. **快速开发**：代码简洁，功能自动生成。
        
    3. **减少Bug**：充分利用编辑器的类型检查，在编码阶段就能发现许多错误。
        
    4. **自描述API**：自动生成交互式 API 文档，前端开发者可以立即使用和测试。
        
    
- **注意事项**：
    
    - FastAPI 需要 Python 3.8+，确保你的开发环境版本正确。
        
    - `uvicorn`是推荐的 ASGI 服务器，用于运行 FastAPI 应用。

# 第二讲：第一个FastAPI应用：5分钟创建"Hello World"

## 概念

**路径操作**是FastAPI中最核心的概念。它指的是"路径"（URL路径，如`/items/foo`）和"HTTP操作"（如GET、POST、PUT、DELETE）的组合。每个路径操作通过**装饰器**与一个处理函数关联，当客户端请求匹配的路径和HTTP方法时，对应的函数就会被执行并返回响应。

## 原理

FastAPI的路径操作系统基于以下工作机制：

1. **路由映射**：当你使用`@app.get("/path")`这样的装饰器时，FastAPI会在内部建立一个路由表，将`("/path", "GET")`映射到你的处理函数。
    
2. **ASGI标准**：FastAPI是基于ASGI（异步服务器网关接口）构建的，这是WSGI的异步演进版本。ASGI允许处理异步请求，支持WebSocket、HTTP/2等现代Web协议。
    
3. **请求-响应周期**：
    
    ```
    客户端请求 → ASGI服务器(如Uvicorn) → FastAPI应用 → 路径操作函数 → 响应序列化 → 返回给客户端
    ```
    
4. **自动序列化**：你的处理函数返回Python对象（字典、列表、Pydantic模型），FastAPI会自动将其转换为JSON格式的HTTP响应。
    

## 例子

让我们创建一个完整的简单API，包含多个路径操作：

```
# 保存为 main.py
from fastapi import FastAPI
from typing import Optional

# 创建FastAPI应用实例
# title、description参数会显示在自动生成的文档中
app = FastAPI(
    title="我的第一个API",
    description="这是一个学习FastAPI的示例应用",
    version="1.0.0"
)

# 定义应用启动时的根路径
@app.get("/")
async def read_root():
    """
    根路径，返回欢迎信息
    
    Returns:
        dict: 包含欢迎消息的字典
    """
    return {
        "message": "欢迎使用FastAPI!",
        "docs": "访问 /docs 查看API文档",
        "redoc": "访问 /redoc 查看ReDoc文档"
    }

# 路径参数示例
@app.get("/users/{user_id}")
async def read_user(user_id: int):
    """
    获取指定ID的用户信息
    
    Args:
        user_id (int): 用户ID，必须是整数
        
    Returns:
        dict: 用户信息
    """
    return {
        "user_id": user_id,
        "name": f"用户{user_id}",
        "action": "获取用户信息成功"
    }

# 查询参数示例
@app.get("/items/")
async def read_items(
    skip: int = 0,      # 默认值为0
    limit: int = 10,    # 默认值为10
    search: Optional[str] = None  # 可选参数
):
    """
    获取物品列表，支持分页和搜索
    
    Args:
        skip (int): 跳过的记录数，用于分页
        limit (int): 返回的最大记录数
        search (str, optional): 搜索关键词
        
    Returns:
        dict: 包含物品列表和查询参数的响应
    """
    # 模拟数据
    all_items = [{"id": i, "name": f"物品{i}"} for i in range(100)]
    
    # 应用搜索过滤
    filtered_items = all_items
    if search:
        filtered_items = [item for item in all_items if search in item["name"]]
    
    # 应用分页
    paginated_items = filtered_items[skip:skip + limit]
    
    return {
        "parameters": {
            "skip": skip,
            "limit": limit,
            "search": search,
            "has_search": search is not None
        },
        "data": paginated_items,
        "total": len(filtered_items),
        "page": skip // limit + 1 if limit > 0 else 1
    }

# POST请求示例
@app.post("/items/")
async def create_item(item: dict):
    """
    创建新物品
    
    Args:
        item (dict): 物品信息
        
    Returns:
        dict: 创建成功的响应
    """
    return {
        "message": "物品创建成功",
        "received_item": item,
        "item_id": 123  # 模拟生成的ID
    }

# PUT和DELETE示例
@app.put("/items/{item_id}")
async def update_item(item_id: int, item: dict):
    return {"message": f"物品{item_id}更新成功", "data": item}

@app.delete("/items/{item_id}")
async def delete_item(item_id: int):
    return {"message": f"物品{item_id}已删除"}

# 复杂的路径参数
@app.get("/files/{file_path:path}")
async def read_file(file_path: str):
    """
    读取文件路径，:path表示匹配完整路径
    
    Args:
        file_path (str): 文件路径
        
    Returns:
        dict: 文件路径信息
    """
    return {"file_path": file_path}
```

运行这个应用：

```
uvicorn main:app --reload --port 8000
```

访问测试：

1. 根路径：`http://127.0.0.1:8000/`
    
2. 用户路径：`http://127.0.0.1:8000/users/42`
    
3. 带查询参数：`http://127.0.0.1:8000/items/?skip=5&limit=3&search=物品`
    
4. 自动文档：`http://127.0.0.1:8000/docs`
    

使用curl或HTTP客户端测试POST请求：

```
curl -X POST "http://127.0.0.1:8000/items/" \
  -H "Content-Type: application/json" \
  -d '{"name": "新物品", "price": 99.99}'
```

## 总结

- **核心要点**：
    
    1. 使用`@app.http_method("路径")`装饰器定义路径操作
        
    2. 支持所有HTTP方法：GET、POST、PUT、DELETE、PATCH、OPTIONS、HEAD
        
    3. 路径参数用`{param}`定义，会作为参数传递给处理函数
        
    4. 查询参数直接在函数参数中定义，可以有默认值
        
    5. 处理函数可以是`async def`（推荐）或普通`def`
        
    
- **路径参数类型**：
    
    - `{user_id}`：普通参数
        
    - `{file_path:path}`：路径参数，可以包含斜杠
        
    
- **最佳实践**：
    
    1. 为`FastAPI()`实例提供`title`、`description`等元数据
        
    2. 为处理函数编写文档字符串，它会显示在自动文档中
        
    3. 使用有意义的函数名和参数名
        
    4. 保持函数专注于单一职责
        
    
- **常见注意事项**：
    
    - 路径操作的顺序很重要：FastAPI按声明顺序匹配路由
        
    - 避免路径冲突，如`/users/me`应该在`/users/{user_id}`之前定义
        
    - 使用类型提示（如`user_id: int`）可以让FastAPI自动验证和转换参数类型
        
    - 对于可选参数，使用`Optional[type] = None`或`type = None`

# 第三讲：深入理解参数：路径参数、查询参数和请求体

## 概念

FastAPI的参数来源主要分为三类，它们共同构成了API的数据入口：

1. **路径参数**：嵌入在URL路径中的变量，例如 `/users/{user_id}`中的 `user_id`。必须提供且不可为空。
    
2. **查询参数**：URL问号后的键值对，例如 `/items?category=book&page=1`中的 `category`和 `page`。通常可选且有默认值。
    
3. **请求体**：HTTP请求正文中的数据，通常是JSON格式。用于POST、PUT等方法传递复杂数据结构。
    

FastAPI通过**类型注解**自动识别参数来源：如果参数在路径中声明了占位符（`{}`），则为路径参数；如果是 `Body`、`Query`、`Path`等显式标注，则强制绑定到对应来源；否则，默认为查询参数（GET等无请求体的方法）或请求体参数（POST等有请求体的方法）。

## 原理

FastAPI的参数解析遵循以下规则：

1. **参数来源推断**：
    
    - 检查函数签名中的参数名是否与路径模板中的变量名匹配。若匹配，则为路径参数。
        
    - 其余参数根据HTTP方法和是否存在 `Body`依赖判断：GET/DELETE等无请求体的方法，未标注的参数视为查询参数；POST/PUT等有请求体的方法，未标注且非路径参数的类型为Pydantic模型的参数视为请求体参数。
        
    
2. **类型验证与转换**：
    
    - 使用Pydantic进行严格的类型检查和转换。例如 `user_id: int`会自动将字符串 `"42"`转为整数 `42`，若无法转换则返回422验证错误。
        
    - 支持泛型、可选类型、枚举、自定义验证器等高级功能。
        
    
3. **依赖注入体系**：
    
    - `Path`、`Query`、`Body`、`Header`、`Cookie`、`Form`、`File`等都是FastAPI提供的“参数依赖项”。它们不仅标记参数来源，还能附加验证规则（如 `Query(max_length=50)`）。
        
    - 当多个参数共用相同验证逻辑时，可使用 `Depends`封装成可复用的依赖函数。
        
    
4. **多源混合**：
    
    - 同一个端点可以同时接收路径参数、查询参数和请求体参数。FastAPI会根据声明自动区分并正确提取。
        
    

## 例子

下面展示一个综合使用三类参数的订单管理端点：

```
from fastapi import FastAPI, Path, Query, Body, HTTPException
from pydantic import BaseModel, Field
from typing import Optional
from enum import Enum

app = FastAPI(title="订单管理系统")

# 枚举类型，限制状态值
class OrderStatus(str, Enum):
    pending = "pending"
    paid = "paid"
    shipped = "shipped"
    delivered = "delivered"
    cancelled = "cancelled"

# Pydantic模型用于请求体验证
class OrderItem(BaseModel):
    product_id: int = Field(..., gt=0, description="商品ID必须大于0")
    quantity: int = Field(..., ge=1, le=100, description="数量1~100")
    unit_price: float = Field(..., gt=0, alias="unitPrice")

    class Config:
        allow_population_by_field_name = True

class CreateOrderRequest(BaseModel):
    customer_name: str = Field(..., min_length=2, max_length=50)
    items: list[OrderItem] = Field(..., min_items=1, max_items=20)
    coupon_code: Optional[str] = None
    notes: Optional[str] = Field(None, max_length=200)

# 路径参数 + 查询参数 + 请求体混合
@app.post("/orders/{store_id}")
async def create_order(
    store_id: int = Path(..., ge=1, description="门店ID"),
    order: CreateOrderRequest = Body(...),
    source: str = Query("web", regex="^(web|mobile|api)$", description="订单来源"),
    priority: bool = Query(False, description="是否加急"),
    debug: bool = Query(False, include_in_schema=False)  # 隐藏于文档
):
    """
    创建订单：路径参数指定门店，查询参数控制行为，请求体包含订单详情
    """
    # 模拟订单处理
    total_amount = sum(
        item.quantity * item.unit_price for item in order.items
    )
    
    # 应用折扣（模拟）
    discount = 0.1 if total_amount > 500 else 0
    
    # 返回完整结果
    return {
        "store_id": store_id,
        "order_summary": {
            "customer": order.customer_name,
            "item_count": len(order.items),
            "total": round(total_amount, 2),
            "discount": discount,
            "final": round(total_amount * (1 - discount), 2)
        },
        "metadata": {
            "source": source,
            "priority": priority,
            "debug_mode": debug
        }
    }

# 另一个端点演示路径参数枚举和查询参数校验
@app.get("/orders/{status}")
async def get_orders_by_status(
    status: OrderStatus = Path(..., description="订单状态"),
    page: int = Query(1, ge=1),
    size: int = Query(20, ge=1, le=100),
    sort_by: str = Query("created_at", regex="^(created_at|total|status)$")
):
    """按状态筛选订单，支持分页和排序"""
    return {
        "status": status.value,
        "pagination": {"page": page, "size": size},
        "sort_by": sort_by,
        "orders": []  # 实际应从数据库查询
    }

# 测试用端点：展示请求头参数（扩展知识）
from fastapi import Header

@app.get("/headers-demo/")
async def demo_headers(
    user_agent: Optional[str] = Header(None),
    x_token: str = Header(..., alias="X-Token")
):
    return {"user_agent": user_agent, "x_token": x_token}
```

## 总结

- **参数来源三剑客**：
    
    - 路径参数：`{param}`，必填，用于标识资源
        
    - 查询参数：`?key=value`，可选或带默认值，用于筛选、分页、配置
        
    - 请求体：JSON正文，用于提交复杂数据（需Pydantic模型）
        
    
- **验证利器**：
    
    - `Path()`、`Query()`、`Body()`等可添加额外约束（如 `ge`、`le`、`regex`、`min_length`）
        
    - Pydantic的 `Field()`提供字段级验证，支持别名、描述等
        
    - 枚举类型自动限制可选值
        
    
- **最佳实践**：
    
    1. 尽量为每个参数添加描述（`description`），提升自动文档质量
        
    2. 敏感或调试参数可用 `include_in_schema=False`隐藏于公开文档
        
    3. 使用 `alias`处理前端传参命名差异（如 `unitPrice`vs `unit_price`）
        
    4. 复杂验证逻辑抽取到 `Depends`依赖函数中复用
        
    
- **常见陷阱**：
    
    - 路径参数与查询参数同名会导致混淆，应避免
        
    - 请求体只能有一个（除非使用多个 `Body`嵌入，但不推荐），复杂数据应嵌套在单个模型中
        
    - 使用 `Optional`时记得设置默认值 `None`，否则仍是必填

# 第四讲：响应模型与数据序列化

## 概念

在FastAPI中，**响应模型**是指定API返回数据结构的方式。通过给路径操作装饰器添加 `response_model`参数，你可以定义返回数据的Pydantic模型，实现以下目标：

1. **数据过滤**：只暴露模型中定义的字段，隐藏敏感信息（如密码、内部ID）。
    
2. **自动序列化**：Python对象（ORM模型、字典、Pydantic实例）自动转换为JSON。
    
3. **类型校验**：确保返回数据符合预期结构，不符合时触发422错误。
    
4. **文档生成**：自动生成OpenAPI响应schema，提升API文档清晰度。
    

**数据序列化**是将内存中的对象（如Pydantic模型实例、SQLAlchemy ORM对象）转换为可传输格式（JSON）的过程。FastAPI基于Pydantic的 `.dict()`和 `.json()`方法实现高效序列化。

## 原理

1. **响应模型的工作流程**：
    
    - 视图函数执行后返回任意对象（Pydantic模型、字典、ORM对象等）。
        
    - FastAPI检查 `response_model`参数，调用Pydantic的 `parse_obj`将原始返回值转换为指定模型实例。
        
    - 如果原始数据包含模型未定义的字段，这些字段会被自动剔除（即“过滤”）。
        
    - 最终通过 `jsonable_encoder`将Pydantic模型转换为JSON兼容的Python字典，再序列化为响应体。
        
    
2. **响应模型与输入模型的区别**：
    
    - 输入模型（请求体）使用 `Body`或继承 `BaseModel`的参数，负责反序列化和验证。
        
    - 响应模型使用 `response_model`参数，负责序列化和过滤。
        
    - 两者可以共享同一个Pydantic模型，但更推荐分离（输入模型含密码，输出模型不含密码）。
        
    
3. **状态码与响应头**：
    
    - 可通过 `status_code`参数固定响应状态码，或使用 `Response`对象动态设置。
        
    - 响应头可通过 `response.headers`或 `Header`依赖项设置。
        
    
4. **嵌套模型与关系**：
    
    - 支持嵌套Pydantic模型（如列表、子模型），自动递归序列化。
        
    - 支持 `orm_mode = True`，允许从ORM对象直接读取属性（无需 `.dict()`）。
        
    

## 例子

下面的例子展示如何构建安全的用户注册API，使用不同的输入/输出模型隐藏密码，并演示多种响应模型技巧。

```
from fastapi import FastAPI, HTTPException, status, Response
from pydantic import BaseModel, EmailStr, Field
from typing import List, Optional
from datetime import datetime

app = FastAPI(title="用户管理系统")

# ---------- 数据模型 ----------
# 输入模型：包含密码字段
class UserCreate(BaseModel):
    username: str = Field(..., min_length=3, max_length=20)
    email: EmailStr
    password: str = Field(..., min_length=8)

# 输出模型：不包含密码，增加id和创建时间
class UserOut(BaseModel):
    id: int
    username: str
    email: str
    created_at: datetime

    class Config:
        orm_mode = True  # 支持从ORM对象读取

# 通用消息模型
class Message(BaseModel):
    detail: str
    code: int

# 用户列表响应
class UsersList(BaseModel):
    users: List[UserOut]
    total: int

# ---------- 模拟数据库 ----------
fake_db = []
next_id = 1

def find_user(username: str) -> Optional[dict]:
    for u in fake_db:
        if u["username"] == username:
            return u
    return None

# ---------- API端点 ----------

# 注册用户：输入模型含密码，输出模型不含密码
@app.post("/users", response_model=UserOut, status_code=status.HTTP_201_CREATED)
async def register_user(user: UserCreate):
    global next_id
    # 检查用户名重复
    if find_user(user.username):
        raise HTTPException(status_code=400, detail="用户名已存在")
    
    new_user = {
        "id": next_id,
        "username": user.username,
        "email": user.email,
        "password": user.password,  # 实际应哈希存储
        "created_at": datetime.now()
    }
    fake_db.append(new_user)
    next_id += 1
    # 返回时自动过滤掉password字段（因为UserOut没有它）
    return new_user  # 返回字典，FastAPI会按UserOut模型过滤

# 获取单个用户：使用响应模型别名
@app.get("/users/{user_id}", response_model=UserOut)
async def get_user(user_id: int):
    for u in fake_db:
        if u["id"] == user_id:
            return u
    raise HTTPException(status_code=404, detail="用户不存在")

# 获取所有用户：返回列表包装模型
@app.get("/users", response_model=UsersList)
async def list_users(skip: int = 0, limit: int = 10):
    # 手动构造响应数据
    users_data = [UserOut(**u) for u in fake_db[skip:skip+limit]]
    return UsersList(users=users_data, total=len(fake_db))

# 演示动态状态码和响应头
@app.post("/messages", response_model=Message, status_code=201)
async def create_message(content: str, response: Response):
    # 设置自定义响应头
    response.headers["X-Message-ID"] = "msg-12345"
    return Message(detail=f"消息 '{content}' 已创建", code=201)

# 演示response_model_exclude_unset和response_model_include
from fastapi import Query

@app.get("/users/partial/{user_id}", 
         response_model=UserOut,
         response_model_exclude_unset=True)  # 只返回显式设置的字段
async def get_partial_user(user_id: int, fields: str = Query("id,username")):
    """
    测试：response_model_exclude_unset=True时，
    如果返回的字典缺少某些字段（如created_at），不会报错，而是忽略未设置字段。
    但这里为了演示，我们始终返回完整数据。
    """
    for u in fake_db:
        if u["id"] == user_id:
            return u
    raise HTTPException(status_code=404)
```

## 总结

- **响应模型的核心作用**：安全过滤、自动序列化、文档生成。
    
- **常用参数**：
    
    - `response_model`：指定输出Pydantic模型。
        
    - `response_model_exclude_unset`：只返回显式赋值的字段（常用于部分更新）。

    3. 复杂嵌套响应使用嵌套Pydantic模型，保持类型安全。
        
    4. 利用 `response_model`的自动文档生成，减少手动编写API文档的工作量。

# 第五讲：路径操作与 HTTP 方法详解

## 概念

**路径操作（Path Operation）**​ 是 FastAPI 对"某条 URL 路径 + 某个 HTTP 方法"组合的统一称呼。比如 `GET /users/me`和 `POST /users`是两个不同的路径操作，哪怕路径部分重叠。

**HTTP 方法**（也叫 HTTP 动词）表达了"对资源做什么"：

- `GET`：获取资源，幂等、无副作用
    
- `POST`：创建新资源
    
- `PUT`：整体替换资源
    
- `PATCH`：部分更新资源
    
- `DELETE`：删除资源
    

RESTful 风格下，API 围绕"资源"设计，URL 是名词（如 `/users`、`/orders`），HTTP 方法是动词。FastAPI 通过 `@app.get()`、`@app.post()`等装饰器直接映射这些方法。

## 原理

1. **路由匹配顺序**：FastAPI 按**代码声明顺序**遍历路由表。所以 `/users/me`必须写在 `/users/{user_id}`之前，否则 `"me"`会被 `{user_id}`捕获，导致 422（因为 `"me"`转不成 `int`）。
    
2. **`status_code`的两种写法**：
    
    - 装饰器参数：`@app.post("/items", status_code=201)`—— 固定状态码
        
    - 返回 `Response`对象动态设置 —— 适合条件分支
        
    - 推荐用 `from fastapi import status`，用 `status.HTTP_201_CREATED`这种常量，可读性好。
        
    
3. **`async def`vs `def`**：
    
    - `async def`：函数内可用 `await`，适合 I/O（查数据库、调外部 API）。FastAPI 在主事件循环里跑它。
        
    - `def`：普通同步函数。FastAPI 会把它扔进线程池跑，避免阻塞事件循环。
        
    - 经验法则：碰 I/O 就用 `async def`；纯 CPU 计算或调同步库（如某些老 ORM）用 `def`也行，但更推荐用 `async def`+ 异步驱动。
        
    
4. **路径参数的"转换器"**：`{file_path:path}`里的 `:path`是 Starlette 的转换器语法，允许匹配含 `/`的段。内置还有 `:int`、`:float`、`:str`（默认）、`:uuid`等。
    

## 例子

用一个"文章管理"的 CRUD 把五种方法串起来，`/articles/me`特意放在 `/articles/{article_id}`前演示顺序坑。

```
from fastapi import FastAPI, HTTPException, status
from pydantic import BaseModel, Field
from typing import List, Optional

app = FastAPI(title="文章管理 API")

# ---------- 模型 ----------
class ArticleCreate(BaseModel):
    title: str = Field(..., min_length=1, max_length=100)
    content: str = Field(..., min_length=1)
    tags: List[str] = Field(default_factory=list)

class ArticleUpdate(BaseModel):
    title: Optional[str] = Field(None, min_length=1, max_length=100)
    content: Optional[str] = None
    tags: Optional[List[str]] = None

class ArticleOut(BaseModel):
    id: int
    title: str
    content: str
    tags: List[str]
    created_at: str  # 简化，用字符串

# ---------- 假数据 ----------
articles = {}
next_id = 1

# ---------- 路径操作 ----------

# 1. GET 列表 —— 无请求体，查询参数做分页/过滤
@app.get("/articles", response_model=List[ArticleOut])
async def list_articles(skip: int = 0, limit: int = 10, tag: Optional[str] = None):
    result = list(articles.values())
    if tag:
        result = [a for a in result if tag in a["tags"]]
    return result[skip : skip + limit]


# 2. GET 单条 —— 路径参数 article_id
# ⚠️ 这条必须放在 /articles/{article_id} 之后，否则 "me" 会被当 id 吞掉
@app.get("/articles/me", response_model=List[ArticleOut])
async def my_articles():
    """当前用户的文章（这里硬编码返回全部，仅演示路由顺序）"""
    return list(articles.values())


@app.get("/articles/{article_id}", response_model=ArticleOut)
async def get_article(article_id: int):
    if article_id not in articles:
        raise HTTPException(status_code=404, detail="文章不存在")
    return articles[article_id]


# 3. POST 创建 —— 201 Created
@app.post("/articles", response_model=ArticleOut, status_code=status.HTTP_201_CREATED)
async def create_article(article: ArticleCreate):
    global next_id
    new = {
        "id": next_id,
        "title": article.title,
        "content": article.content,
        "tags": article.tags,
        "created_at": "2026-01-15T10:00:00Z"
    }
    articles[next_id] = new
    next_id += 1
    return new


# 4. PUT 整体替换
@app.put("/articles/{article_id}", response_model=ArticleOut)
async def replace_article(article_id: int, article: ArticleCreate):
    if article_id not in articles:
        raise HTTPException(status_code=404, detail="文章不存在")
    articles[article_id] = {
        "id": article_id,
        "title": article.title,
        "content": article.content,
        "tags": article.tags,
        "created_at": articles[article_id]["created_at"]  # 保留原创建时间
    }
    return articles[article_id]


# 5. PATCH 部分更新
@app.patch("/articles/{article_id}", response_model=ArticleOut)
async def update_article(article_id: int, patch: ArticleUpdate):
    if article_id not in articles:
        raise HTTPException(status_code=404, detail="文章不存在")
    stored = articles[article_id]
    update_data = patch.dict(exclude_unset=True)  # 只取传了的字段
    stored.update(update_data)
    return stored


# 6. DELETE
@app.delete("/articles/{article_id}", status_code=status.HTTP_204_NO_CONTENT)
async def delete_article(article_id: int):
    if article_id not in articles:
        raise HTTPException(status_code=404, detail="文章不存在")
    del articles[article_id]
    # 204 通常不返回 body，FastAPI 会处理成空响应
    return
```

```

  -H 'Content-Type: application/json' \

# 列表
curl http://127.0.0.1:8000/articles?tag=python

# 部分更新

curl -X DELETE -i http://127.0.0.1:8000/articles/1
```

## 总结

- **RESTful 语义对应**：GET 查 / POST 建 / PUT 全换 / PATCH 部改 / DELETE 删，URL 用名词复数（`/articles`），不用动词（`/getArticle`）。
    
- **路由顺序坑**：`/articles/me`必须声明在 `/articles/{article_id}`之前，否则 `me`被当路径参数吞掉，报 422。
    
- **`status_code`用 `fastapi.status`常量**：`HTTP_201_CREATED`、`HTTP_204_NO_CONTENT`、`HTTP_404_NOT_FOUND`比裸写数字可读性强得多。
    
- **`async def`是默认推荐**：除非你明确需要同步阻塞调用，否则都用 `async def`；FastAPI 对 `def`会丢进线程池，但不是首选。
    
- **PATCH + `exclude_unset=True`是部分更新的黄金搭档**：前端只传想改的字段，后端只更新这些字段，不覆盖没传的。

# 第六讲：错误处理与自定义异常

## 概念

FastAPI 的错误处理分三层：

1. **`HTTPException`**：最常用，在路径操作函数里直接 `raise`，FastAPI 捕获后返回对应的 HTTP 响应（状态码 + JSON body `{"detail": "..."}`）。
    
2. **自定义异常 + `@app.exception_handler()`**：自己定义一个业务异常类（如 `BusinessError`），再用 `@app.exception_handler(BusinessError)`注册全局处理器，统一把异常转成标准响应格式。
    
3. **FastAPI 内置异常的覆写**：比如请求体验证失败默认返回 422 + 一堆 validation error 数组，可以用 `exception_handler(RequestValidationError)`换成自己想要的格式。
    

企业项目里通常会做**统一响应信封**（如 `{"code": 0, "msg": "ok", "data": ...}`），错误时也保持同一结构（`{"code": 400xx, "msg": "...", "data": null}`），前端好处理。这一讲就围绕这个目标展开。

## 原理

1. **`HTTPException`的流程**：
    
    - 你在视图里 `raise HTTPException(status_code=404, detail="用户不存在")`
        
    - Starlette 的异常中间件捕获它，调 `HTTPException`自带的 `handler`，返回 JSON `{"detail": "用户不存在"}`+ 对应状态码
        
    - `headers`参数还能额外加响应头（比如 401 时加 `WWW-Authenticate`）
        
    
2. **`@app.exception_handler(异常类)`的机制**：
    
    - FastAPI/Starlette 有一张"异常类型 → 处理函数"的映射表
        
    - 请求过程中抛出的异常，按 MRO（方法解析顺序）找最匹配的 handler
        
    - 所以自定义异常继承链设计好，就能分层捕获（比如 `BusinessError`基类 → `NotFoundError( BusinessError)`子类）
        
    
3. **`RequestValidationError`的覆写要点**：
    
    - 默认 handler 在 `fastapi.exception_handlers`里，返回 422 + `{"detail": [...]}`（数组里每个元素是 field/path + msg + type）
        
    - 覆写时拿到 `RequestValidationError`对象，`.errors()`方法给出结构化错误信息，你可以自己拼格式
        
    
4. **`@app.middleware("http")`也能拦异常吗？**​ —— 不能。中间件在异常处理器**外层**，如果异常被 handler 处理了，中间件看到的是正常响应；如果没被 handle 掉（编程 bug 抛了普通 `Exception`），才会冒到 ASGI 层返回 500。所以"全局兜底 500"得用 `@app.exception_handler(Exception)`或 Starlette 的 `ServerErrorMiddleware`。
    

## 例子

延续前面几讲的"文章管理"语境，加上统一响应信封和三层错误处理。

```
from fastapi import FastAPI, HTTPException, Request, status
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from pydantic import BaseModel, Field, ValidationError
from typing import Any, Optional, List
from datetime import datetime

app = FastAPI(title="文章管理 API — 错误处理篇")

# ========== 统一响应信封 ==========

def success_response(data: Any = None, msg: str = "ok"):
    return {"code": 0, "msg": msg, "data": data}

def error_response(code: int, msg: str, data: Any = None):
    return {"code": code, "msg": msg, "data": data}

# ========== 自定义业务异常 ==========

class BusinessError(Exception):
    """业务异常基类"""
    def __init__(self, code: int, msg: str, status_code: int = 400):
        self.code = code       # 业务码，如 40001
        self.msg = msg         # 业务消息
        self.status_code = status_code  # HTTP 状态码
        super().__init__(msg)

class NotFoundError(BusinessError):
    def __init__(self, resource: str = "资源"):
        super().__init__(code=40401, msg=f"{resource}不存在", status_code=404)

class ForbiddenError(BusinessError):
    def __init__(self, msg: str = "无权限"):
        super().__init__(code=40301, msg=msg, status_code=403)

# ========== 全局异常处理器 ==========

@app.exception_handler(BusinessError)
async def business_error_handler(request: Request, exc: BusinessError):
    """所有 BusinessError 子类都会被这里兜住"""
    return JSONResponse(
        status_code=exc.status_code,
        content=error_response(exc.code, exc.msg)
    )

@app.exception_handler(RequestValidationError)
async def validation_error_handler(request: Request, exc: RequestValidationError):
    """覆写 422 的默认格式，换成统一信封"""
    # exc.errors() 是 pydantic 的错误列表，每个有 loc / msg / type
    first = exc.errors()[0]
    loc = ".".join(str(x) for x in first["loc"])
    msg = f"{loc}: {first['msg']}"
    return JSONResponse(
        status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
        content=error_response(42201, msg, data=exc.errors())
    )

@app.exception_handler(Exception)
async def unhandled_exception_handler(request: Request, exc: Exception):
    """兜底：代码 bug 漏网的普通 Exception，返回 500"""
    # 生产环境这里应该打日志，不要把 exc 细节返给前端
    return JSONResponse(
        status_code=500,
        content=error_response(50000, "服务器内部错误")
    )

# ========== 模型 ==========

class ArticleCreate(BaseModel):
    title: str = Field(..., min_length=1, max_length=100)
    content: str = Field(..., min_length=1)

# 假数据
articles = {1: {"id": 1, "title": "FastAPI 入门", "content": "..."}}

# ========== 端点 ==========

# 1. 用 HTTPException（简单场景够用）
@app.get("/articles/{article_id}")
async def get_article(article_id: int):
    if article_id not in articles:
        # HTTPException 是最轻量的报错方式
        raise HTTPException(status_code=404, detail="文章不存在")
    return success_response(articles[article_id])

# 2. 用自定义 BusinessError（推荐，业务码更清晰）
@app.get("/articles/biz/{article_id}")
async def get_article_biz(article_id: int):
    if article_id not in articles:
        raise NotFoundError("文章")
    return success_response(articles[article_id])

# 3. 触发验证错误，看覆写效果
@app.post("/articles")
async def create_article(article: ArticleCreate):
    # 这里故意不动 articles，只看验证
    return success_response(article.dict() | {"id": 999})

# 4. 触发 500 兜底
@app.get("/crash")
async def crash():
    """故意抛个普通 Exception，看兜底 handler"""
    1 / 0  # ZeroDivisionError
    return {"ok": True}
```

### 用 curl 看效果

```
# 1. HTTPException 默认格式（FastAPI 原生，不在 handler 里兜）
curl http://127.0.0.1:8000/articles/99
# → 404, {"detail": "文章不存在"}  （HTTPException 没走 BusinessError handler）

# 2. 自定义 BusinessError → 统一信封
curl http://127.0.0.1:8000/articles/biz/99
# → 404, {"code": 40401, "msg": "文章不存在", "data": null}

# 3. 验证错误覆写
curl -X POST http://127.0.0.1:8000/articles \
  -H 'Content-Type: application/json' \
  -d '{"title":"","content":""}'
# → 422, {"code": 42201, "msg": "title: ensure this value has at least 1 character", ...}

# 4. 500 兜底
curl http://127.0.0.1:8000/crash
# → 500, {"code": 50000, "msg": "服务器内部错误"}
```

> 💡 细心的会发现：例 1 的 `HTTPException`返回格式还是 `{"detail": "..."}`，没有被 `error_response`包住。这是因为 `HTTPException`是 Starlette 层的异常，FastAPI 有自己的默认 handler，不在我们写的 `BusinessError`handler 覆盖范围内。想让 `HTTPException`也走统一信封，有两种办法：① 别直接 raise `HTTPException`，封装一个 `raise Biz(code=..., msg=..., http_status=...)`统一走 `BusinessError`；② 再补一个 `@app.exception_handler(HTTPException)`把 `HTTPException`也转成统一信封。企业项目里我推荐方案①，少一层 handler 更干净。

## 总结

- **三层错误处理**：
    
    - `HTTPException`：快速抛出 HTTP 错误，适合简单场景，但格式固定为 `{"detail": "..."}`
        
    - `自定义异常 + @app.exception_handler`：业务码 + 统一信封，企业项目首选
        
    - `RequestValidationError`覆写：把 422 默认格式换成自己的，前端一致性更好
        
    
- **`@app.exception_handler(Exception)`兜底**：所有没被处理的异常最后到这里，返回 500。生产环境务必打日志，不要把堆栈返前端。
    
- **统一响应信封的价值**：前端不用针对 200/422/404 写不同解析逻辑，`code=0`成功、非 0 看 `msg`，一条规则通吃。
    
- **常见坑**：
    
    - `HTTPException`和自定义 `BusinessError`并存时，格式会不一致 → 建议项目里只选一种风格
        
    - 验证错误 `exc.errors()`的结构是 Pydantic 定的，前端如果想直接消费，可以整个 `data`透出去，也可以像上面只取 `first`拼一句话
        
    - 中间件拦不到已被 handler 处理掉的异常，兜底 500 得靠 `exception_handler(Exception)`

# 第七讲：依赖注入与中间件

## 概念

**依赖注入（Dependency Injection, DI）**​ 是 FastAPI 最强大的特性之一。它让你把公共逻辑（如数据库会话、当前用户、分页参数、权限校验）抽离成独立的函数或类，然后在路径操作函数中用 `Depends()`声明"我需要这个依赖"，FastAPI 会自动调用依赖函数并把结果注入进来。

**中间件（Middleware）**​ 是另一种横切关注点的处理方式。它是一个钩子，在每个请求进入路由之前、响应返回客户端之前执行。适合做全局性的工作，比如记录请求耗时、设置 CORS 头、压缩响应、IP 黑名单等。

两者的核心区别：

- **Depends**​ 是**函数级别的注入**，粒度细，可以在不同路径操作中选择性地使用，也能访问路径操作的参数。
    
- **中间件**​ 是**全局的管道**，对所有请求生效（除非按路径跳过），无法直接访问路径操作的特定参数。
    

实际项目中，二者经常配合使用：中间件做全局预处理（如日志、CORS），Depends 做业务相关的复用（如鉴权、数据库会话）。

## 原理

1. **`Depends()`的工作机制**：
    
    - 当你写 `def endpoint(db: Session = Depends(get_db))`时，FastAPI 会在路由注册阶段分析函数的签名，发现 `db`参数有 `Depends(get_db)`作为默认值。
        
    - 请求到来时，FastAPI 先调用 `get_db()`获取返回值，然后把这个返回值赋给 `db`参数。
        
    - 依赖函数本身也可以有参数（包括其他 `Depends`），形成**依赖树**。FastAPI 会按拓扑序解析所有依赖，确保每个依赖只执行一次（即使被多次引用）。
        
    - 依赖可以是函数、类、生成器（用于需要清理资源的场景，如数据库连接关闭）。
        
    
2. **中间件的工作机制**：
    
    - 中间件是一个可调用对象，接收 `request`和一个 `call_next`函数。
        
    - `call_next(request)`会将请求传递给下一个中间件或路由处理器，并返回响应。
        
    - 你可以在调用 `call_next`之前做请求预处理（如计时开始），之后做响应后处理（如添加头、记录耗时）。
        
    - 中间件按添加顺序组成链，`@app.middleware("http")`注册的函数会依次执行。
        
    
3. **Depends 与中间件的协作**：
    
    - 中间件无法直接向路径操作注入参数（它拿不到 `Depends`的结果），但可以通过 `request.state`传递数据。例如中间件解析 token 后设置 `request.state.user`，然后在依赖函数中通过 `request: Request = Depends()`读取。
        
    - 反过来，Depends 也不能拦截请求（它只是提供参数），而中间件可以提前终止请求（如返回 403）。
        
    

## 例子

延续文章管理 API，加入认证依赖、分页依赖、数据库会话依赖（模拟），以及一个记录耗时的中间件。

```
import time
from fastapi import FastAPI, Depends, HTTPException, Request, status
from fastapi.responses import JSONResponse
from pydantic import BaseModel, Field
from typing import List, Optional, Generator

app = FastAPI(title="文章管理 — 依赖注入与中间件")

# ========== 模拟数据库会话 ==========
class DBSession:
    """模拟数据库会话，实际用 SQLAlchemy 的 Session"""
    def __init__(self):
        self.connected = True

    def close(self):
        self.connected = False

# 生成器形式的依赖：在请求结束后自动关闭会话
def get_db() -> Generator[DBSession, None, None]:
    db = DBSession()
    try:
        yield db
    finally:
        db.close()

# ========== 认证依赖 ==========
# 假设从 Header 中读取 X-Token 来验证用户
def get_current_user(request: Request) -> dict:
    token = request.headers.get("X-Token")
    if not token:
        raise HTTPException(status_code=401, detail="缺少 Token")
    # 模拟验证，实际应查数据库或 JWT 解码
    if token != "valid-token":
        raise HTTPException(status_code=403, detail="无效 Token")
    return {"username": "alice", "role": "admin"}

# 可选的当前用户（匿名也可访问）
def get_optional_user(request: Request) -> Optional[dict]:
    token = request.headers.get("X-Token")
    if token == "valid-token":
        return {"username": "alice", "role": "admin"}
    return None

# ========== 分页依赖 ==========
class Pagination(BaseModel):
    skip: int = 0
    limit: int = 10

def get_pagination(skip: int = 0, limit: int = 10) -> Pagination:
    return Pagination(skip=skip, limit=limit)

# ========== 权限校验依赖（依赖其他依赖） ==========
def require_admin(current_user: dict = Depends(get_current_user)) -> dict:
    if current_user["role"] != "admin":
        raise HTTPException(status_code=403, detail="需要管理员权限")
    return current_user

# ========== 中间件：记录请求耗时 ==========
@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start_time = time.time()
    response = await call_next(request)
    process_time = time.time() - start_time
    response.headers["X-Process-Time"] = str(round(process_time, 4))
    return response

# ========== 模型 ==========
class ArticleOut(BaseModel):
    id: int
    title: str
    content: str

# 假数据
articles_db = {
    1: {"id": 1, "title": "FastAPI 入门", "content": "..."},
    2: {"id": 2, "title": "依赖注入实战", "content": "..."}
}

# ========== 路径操作 ==========

# 1. 需要登录才能查看文章列表（使用 get_current_user 依赖）
@app.get("/articles", response_model=List[ArticleOut])
async def list_articles(
    pagination: Pagination = Depends(get_pagination),
    current_user: dict = Depends(get_current_user),
    db: DBSession = Depends(get_db)
):
    """需要认证，支持分页"""
    all_articles = list(articles_db.values())
    return all_articles[pagination.skip : pagination.skip + pagination.limit]

# 2. 公开接口：无需登录，但有可选用户信息
@app.get("/public/articles", response_model=List[ArticleOut])
async def public_articles(
    user: Optional[dict] = Depends(get_optional_user)
):
    if user:
        print(f"用户 {user['username']} 访问了公开文章")
    return list(articles_db.values())

# 3. 管理员专属：需要 admin 权限
@app.post("/admin/reset-database")
async def reset_database(
    admin: dict = Depends(require_admin)
):
    """只有 admin 能重置数据库"""
    articles_db.clear()
    return {"message": "数据库已重置", "operator": admin["username"]}

# 4. 演示依赖树：一个依赖依赖另一个依赖
@app.get("/profile")
async def get_profile(
    current_user: dict = Depends(get_current_user),
    db: DBSession = Depends(get_db)
):
    """查看个人信息（依赖认证和数据库会话）"""
    return {
        "user": current_user,
        "db_connected": db.connected
    }
```

## 总结

- **Depends 是 FastAPI 依赖注入的核心**：可复用、可组合、支持生成器清理资源、支持依赖树。
    
- **典型用途**：
    
    - 数据库会话（`yield`模式自动关闭）
        
    - 当前用户认证（从 Header/Cookie/JWT 解析）
        
    - 分页/排序参数封装
        
    - 权限校验（依赖其他依赖）
        
    
- **中间件做全局横切关注点**：
    
    - 请求耗时统计、CORS、GZip 压缩、日志记录
        
    - 不能在中间件里直接给路径操作注入参数，但可以通过 `request.state`传递
        
    
- **Depends 与中间件的选择**：
    
    - 需要在多个路径操作中**选择性使用**​ → Depends
        
    - 需要对**所有请求**生效且不关心具体业务参数 → 中间件
        
    - 既需要全局又需要访问业务参数 → 中间件 + `request.state`+ Depends 读取
        
    
- **常见陷阱**：
    
    - 依赖函数不要定义为 `async`除非你真的需要 `await`，否则同步即可，FastAPI 会自动在线程池运行。
        
    - 生成器依赖的 `finally`块在请求结束后执行，确保资源释放（如关闭数据库连接）。
        
    - 不要在依赖中修改 `request`的不可变部分（如 `request.path_params`），但可以安全地设置 `request.state`。

# 第八讲：WebSocket 实时通信

## 概念

**WebSocket**​ 是一种在单个 TCP 连接上实现**全双工通信**的协议，不同于 HTTP 的"一问一答"，WebSocket 在客户端和服务端建立连接后可以**双向持续收发消息**，直到任意一方主动关闭。

FastAPI 基于 Starlette 提供了对 WebSocket 的原生支持，API 风格和路径操作高度相似：用 `@app.websocket("/path")`装饰器定义端点，在处理器里用 `await websocket.accept()`握手，然后用 `await websocket.receive_text()`/ `receive_json()`收消息，`await websocket.send_text()`/ `send_json()`发消息。

典型使用场景：实时聊天、行情推送、日志流、在线协作（如多人同时编辑文档）、游戏状态同步。

## 原理

1. **协议升级流程**：
    
    - 客户端发起 `GET /chat`请求，带上特殊的 `Upgrade: websocket`+ `Connection: Upgrade`+ `Sec-WebSocket-Key`头
        
    - 服务端回应 `101 Switching Protocols`，双方从此脱离 HTTP，走 WebSocket 帧
        
    - FastAPI/Starlette 帮你封装了握手过程，你只需要调 `websocket.accept()`
        
    
2. **WebSocket 端点的结构**：
    
    ```
    @app.websocket("/ws")
    async def websocket_endpoint(websocket: WebSocket):
        await websocket.accept()
        try:
            while True:
                data = await websocket.receive_text()
                await websocket.send_text(f"你说: {data}")
        except WebSocketDisconnect:
            # 客户端断开
            pass
    ```
    
    - `WebSocket`对象是 Starlette 提供的，包含 `accept()`、`receive_text()`、`send_text()`、`close()`等方法
        
    - 通常用 `while True`循环持续收消息，客户端断开时会抛 `WebSocketDisconnect`异常，捕获即可做清理
        
    
3. **依赖注入在 WebSocket 中的用法**：
    
    - WebSocket 端点**不能直接用**​ `Depends`作为函数参数（Starlette 的 WebSocket 路由签名和 HTTP 不同），但可以在函数体内手动调依赖，或通过 `websocket.query_params`/ `websocket.headers`取 token 做鉴权
        
    - 更优雅的做法：用 `WebSocketEndpoint`类（Starlette 提供）或 FastAPI 的 `@app.websocket`+ 在函数开头手动调 `get_current_user_from_ws(websocket)`这种包装函数
        
    
4. **连接管理**：
    
    - 要支持"广播"（如聊天室），需要维护一个**活跃连接列表**，新连接 append，断开时 remove
        
    - 多进程/多机部署时，内存里的连接列表不够用，需要 Redis Pub/Sub 或类似方案——这个超出单讲范围，点到为止
        
    

## 例子

做一个简易聊天室 + 一个实时日志流端点，覆盖连接、收发、广播、断开、鉴权。

```
import asyncio
import json
from fastapi import FastAPI, WebSocket, WebSocketDisconnect, Depends, Query
from fastapi.responses import HTMLResponse
from typing import List, Dict

app = FastAPI(title="WebSocket 演示")

# ========== 简易聊天室 ==========

class ConnectionManager:
    """管理所有活跃 WebSocket 连接（广播场景）"""
    def __init__(self):
        self.active_connections: List[WebSocket] = []

    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)

    def disconnect(self, websocket: WebSocket):
        self.active_connections.remove(websocket)

    async def send_personal(self, message: str, websocket: WebSocket):
        await websocket.send_text(message)

    async def broadcast(self, message: str):
        for conn in self.active_connections:
            await conn.send_text(message)

manager = ConnectionManager()

# 聊天室端点
@app.websocket("/ws/chat/{username}")
async def websocket_chat(websocket: WebSocket, username: str):
    """
    路径参数 username 直接嵌在 WS URL 里，简单鉴权
    实际项目应该从 query token 解码，这里简化
    """
    await manager.connect(websocket)
    await manager.broadcast(f"📢 {username} 加入了聊天室")

    try:
        while True:
            data = await websocket.receive_text()
            # 广播给所有人：xxx 说 yyy
            await manager.broadcast(f"{username}: {data}")
    except WebSocketDisconnect:
        manager.disconnect(websocket)
        await manager.broadcast(f"📢 {username} 离开了聊天室")

# ========== 实时日志流（服务端主动推送） ==========

@app.websocket("/ws/logs")
async def websocket_logs(
    websocket: WebSocket,
    interval: float = Query(1.0, ge=0.1, le=10.0)  # 推送间隔，秒
):
    """
    模拟实时日志流：服务端每隔 interval 秒主动推一条日志
    客户端只管收，不发消息也没关系
    """
    await websocket.accept()
    try:
        count = 0
        while True:
            count += 1
            log = {
                "timestamp": count,
                "level": "INFO" if count % 3 != 0 else "WARN",
                "message": f"模拟日志条目 #{count}"
            }
            await websocket.send_json(log)
            await asyncio.sleep(interval)
    except WebSocketDisconnect:
        pass  # 客户端断开，循环自然结束

# ========== 带简单鉴权的 WS（从 query 取 token） ==========

async def verify_ws_token(
    websocket: WebSocket,
    token: str = Query(...)
) -> dict:
    """WS 鉴权：从 query 参数取 token 验证"""
    if token != "secret-ws-token":
        # 鉴权失败：不 accept，直接 close，客户端会收到 403 类断开
        await websocket.close(code=4003, reason="Invalid token")
        # 抛异常让端点函数不继续执行
        raise WebSocketDisconnect(code=4003)
    return {"user": "alice", "token": token}

@app.websocket("/ws/protected")
async def websocket_protected(
    websocket: WebSocket,
    user: dict = Depends(verify_ws_token)  # 注意：这里 Depends 能用！
):
    """
    FastAPI 对 WebSocket 端点也支持 Depends（新版本特性）
    验证通过后 user 被注入
    """
    await websocket.accept()
    await websocket.send_json({"msg": f"欢迎, {user['user']}!", "status": "connected"})
    try:
        while True:
            data = await websocket.receive_text()
            await websocket.send_text(f"回声: {data}")
    except WebSocketDisconnect:
        pass

# ========== 一个简单的聊天室前端页面（方便测试） ==========

@app.get("/chat-room")

    html = """
```

3. **鉴权 WS**：
    
    ```
    wscat -c "ws://127.0.0.1:8000/ws/protected?token=secret-ws-token"
    # 换错 token 试试：wscat -c "ws://127.0.0.1:8000/ws/protected?token=wrong"  → 会断开
    ```
    

## 总结

- **WebSocket 端点写法**：`@app.websocket("/path")`+ `async def endpoint(ws: WebSocket)`，握手 `await ws.accept()`，循环 `receive_text()`/ `send_text()`，捕获 `WebSocketDisconnect`做清理。
    
- **广播模式**：用 `ConnectionManager`维护活跃连接列表，`connect`时 append，`disconnect`时 remove，`broadcast`时遍历 send。**单进程 OK，多进程需 Redis Pub/Sub**。
    
- **鉴权两种方式**：
    
    - 路径参数 `/ws/chat/{username}`—— 简单但不安全，仅演示
        
    - Query 参数 `?token=xxx`+ 手动 `verify_ws_token`—— 推荐，能在 accept 前拒绝
        
    - FastAPI 新版本对 WS 端点也支持 `Depends`（如上例 `/ws/protected`），但注意老版本不支持，稳妥起见可在函数体内手动调
        
    
- **`receive_*`家族**：`receive_text()`/ `receive_json()`/ `receive_bytes()`，对应 `send_text()`/ `send_json()`/ `send_bytes()`
    
- **常见坑**：
    
    - 忘了 `await websocket.accept()`就直接 `receive/send`→ 报错
        
    - 客户端断开后 `receive_text()`抛 `WebSocketDisconnect`，必须捕获，否则整个端点炸
        
    - 长循环里没 `try/except WebSocketDisconnect`→ 日志刷满
        
    - 广播时某个连接已死但还在列表里 → `send`会抛异常，最好在 broadcast 里包 `try`单条失败就 `disconnect`它

# 第九讲：Form 表单、文件上传与 `multipart/form-data`

## 概念

HTTP 表单提交主要有两种编码类型：

- **`application/x-www-form-urlencoded`**：键值对格式，适合纯文本字段，浏览器默认方式。
    
- **`multipart/form-data`**：二进制分界格式，每个字段（包括文件）用 boundary 分隔，适合上传文件或混合表单。
    

FastAPI 对这两种都提供原生支持：

- **`Form`**：声明表单字段，等价于 `Query`/`Path`的用法，但数据来自请求体（而非 URL）。
    
- **`File`**​ / **`UploadFile`**：声明文件上传字段。`File`返回字节内容（`bytes`），`UploadFile`返回一个类似文件的 SpooledTemporaryFile 对象，提供文件名、content_type、异步读写方法，更适合大文件。
    

## 原理

1. **`Form`的工作原理**：
    
    - 当参数类型是 `Form(...)`时，FastAPI 会从请求体中解析 `multipart/form-data`或 `application/x-www-form-urlencoded`格式。
        
    - 与 `Body`的区别：`Body`期望 JSON，`Form`期望表单编码。如果混用，FastAPI 会报错（同一个端点不能同时接受 JSON 和 Form，除非用 `Body(media_type="...")`特殊处理，但很少这样做）。
        
    
2. **`File`与 `UploadFile`的选择**：
    
    - `File(...)`：直接读取文件全部内容到内存，返回 `bytes`。适合小文件（几 MB 以内），简单直接。
        
    - `UploadFile(...)`：返回 Starlette 的 `UploadFile`对象，文件内容暂存于临时文件中（超过阈值才写入磁盘）。提供 `.read()`、`.write()`、`.seek()`、`.filename`、`.content_type`等属性和方法。适合大文件或需要流式处理的场景。
        
    
3. **多文件上传**：
    
    - 使用 `List[UploadFile] = File(...)`可以接收多个同名文件字段（HTML `<input type="file" multiple>`）。
        
    - 或者多个不同名字段分别声明。
        
    
4. **混合表单（字段 + 文件）**：
    
    - 可以在同一个端点中同时使用 `Form`和 `File`/`UploadFile`。FastAPI 会自动将它们从 `multipart/form-data`中提取出来。
        
    
5. **文件保存**：
    
    - FastAPI 不负责持久化文件，你需要自己写文件到磁盘或云存储。示例中会用 `aiofiles`异步写入本地目录。
        
    

## 例子

实现一个用户头像上传接口和一个多文件上传接口，覆盖纯字段表单、单文件、多文件、混合表单。

```
import os
import shutil
from pathlib import Path
from fastapi import FastAPI, Form, File, UploadFile, HTTPException, Depends
from fastapi.responses import JSONResponse
from typing import List, Optional
import aiofiles  # 需要 pip install aiofiles

app = FastAPI(title="文件上传演示")

UPLOAD_DIR = Path("./uploads")
UPLOAD_DIR.mkdir(exist_ok=True)

# ========== 1. 纯表单字段（无文件） ==========

@app.post("/form-only")
async def handle_form(
    username: str = Form(..., min_length=3, max_length=20),
    age: int = Form(..., ge=0, le=150),
    bio: Optional[str] = Form(None, max_length=500)
):
    """仅接收表单字段，无文件"""
    return {
        "username": username,
        "age": age,
        "bio": bio or "(未填写)"
    }

# ========== 2. 单文件上传（使用 UploadFile） ==========

@app.post("/upload/avatar")
async def upload_avatar(
    avatar: UploadFile = File(..., description="头像图片，建议小于5MB"),
    user_id: str = Form(...)  # 混合表单字段
):
    # 验证文件类型
    allowed_types = ["image/jpeg", "image/png", "image/gif"]
    if avatar.content_type not in allowed_types:
        raise HTTPException(400, f"不支持的文件类型: {avatar.content_type}，仅支持 JPEG/PNG/GIF")

    # 生成唯一文件名
    ext = Path(avatar.filename).suffix if avatar.filename else ".jpg"
    file_path = UPLOAD_DIR / f"avatar_{user_id}{ext}"

    # 异步写入文件
    async with aiofiles.open(file_path, "wb") as f:
        # 分块读取，避免大文件撑爆内存
        while chunk := await avatar.read(1024 * 1024):  # 每次读1MB
            await f.write(chunk)

    return {
        "message": "头像上传成功",
        "filename": avatar.filename,
        "saved_as": str(file_path),
        "size_hint": "已保存到服务器"
    }

# ========== 3. 多文件上传（同名字段） ==========

@app.post("/upload/photos")
async def upload_photos(
    photos: List[UploadFile] = File(..., description="选择多张照片"),
    album: str = Form("default")
):
    """上传多张照片到相册"""
    saved_files = []
    for photo in photos:
        if photo.filename is None:
            continue
        file_path = UPLOAD_DIR / f"{album}_{photo.filename}"
        async with aiofiles.open(file_path, "wb") as f:
            while chunk := await photo.read(1024 * 512):
                await f.write(chunk)
        saved_files.append({
            "original": photo.filename,
            "saved_as": str(file_path),
            "content_type": photo.content_type
        })

    return {
        "message": f"成功上传 {len(saved_files)} 张照片",
        "album": album,
        "files": saved_files
    }

# ========== 4. 使用 File() 返回 bytes（小文件） ==========

@app.post("/upload/small-file")
async def upload_small_file(
    file: bytes = File(..., description="小文件（建议<1MB）"),
    note: str = Form("")
):
    """直接接收文件内容为 bytes，适合小文件"""
    size = len(file)
    return {
        "note": note,
        "size_bytes": size,
        "md5_preview": hashlib.md5(file).hexdigest()[:16]  # 需要 import hashlib
    }
```

## 总结

- **表单编码类型**：
    
    - `application/x-www-form-urlencoded`：纯文本键值对，适合简单表单
        
    - `multipart/form-data`：支持文件和二进制数据，必须用于文件上传
        
    
- **FastAPI 声明方式**：
    
    - `Form(...)`：声明表单字段，支持验证（`min_length`、`ge`等）
        
    - `File(...)`：返回 `bytes`，适合小文件
        
    - `UploadFile(...)`：返回 `UploadFile`对象，适合大文件或需要文件名/content_type
        
    - `List[UploadFile] = File(...)`：多文件上传
        
    
- **重要限制**：
    
    - 同一个端点**不能同时使用 `Body`（JSON）和 `Form`/`File`**，必须二选一。如果需要混合 JSON 和文件，考虑用 Base64 编码文件放入 JSON，或拆分成两个端点。
        
    - 文件大小受限于 FastAPI/Starlette 的配置（默认无上限，但生产环境应在反向代理层限制）。
        
    
- **最佳实践**：
    
    - 始终验证文件类型（`content_type`不可完全信任，但可作为第一道防线）。
        
    - 使用 `aiofiles`异步写入文件，避免阻塞事件循环。
        
    - 分块读取（`await read(size)`）防止大文件撑爆内存。
        
    - 生成唯一文件名（UUID + 扩展名）避免冲突和路径穿越攻击。
        
    - 上传目录不要暴露在静态文件服务下，或严格控制权限。

# 第十讲：FastAPI 测试——用 TestClient 做单元测试与集成测试

## 概念

FastAPI 基于 Starlette 提供了一个开箱即用的测试工具 **`TestClient`**，它继承自 `httpx`的 `AsyncClient`，让你可以不启动真实服务器，直接在 Python 代码中模拟 HTTP 请求，测试路径操作的全部环节：参数解析、依赖注入、错误处理、响应模型。

测试分为两类：

- **单元测试**：测试单个路径操作函数的逻辑，可以 mock 外部依赖（如数据库）。
    
- **集成测试**：测试多个组件协同工作的流程，通常使用真实的测试数据库。
    

`TestClient`既可以同步使用（`with TestClient(app) as client:`），也可以异步使用（需要安装 `httpx`的异步支持）。对于大多数测试场景，同步模式已经足够。

## 原理

1. **`TestClient`的工作方式**：
    
    - 它在内部创建一个 ASGI 实例，直接调用 FastAPI 应用的 `__call__`方法，不走网络 socket。
        
    - 因此测试速度极快，且不需要单独启动 uvicorn 进程。
        
    - 请求和响应的接口与 `requests`库几乎一致（因为 httpx 兼容 requests API）。
        
    
2. **测试夹具（Fixture）**：
    
    - pytest 的 fixture 机制非常适合准备测试数据、初始化依赖。
        
    - 可以将 `TestClient`本身作为一个 fixture，在测试函数间复用。
        
    
3. **覆盖依赖注入**：
    
    - 测试时可以替换真实依赖（如数据库会话）为 mock 对象，通过 `app.dependency_overrides`字典实现。
        
    - 这是 FastAPI 测试的核心优势：不修改任何业务代码，就能替换依赖。
        
    
4. **断言响应**：
    
    - 检查状态码、响应头、响应体 JSON 结构。
        
    - 使用 Pydantic 模型验证响应数据是否符合预期 schema。

### 关键点说明

1. **`app.dependency_overrides`**：在测试中替换依赖的"魔法字典"。`override_get_db`返回一个假数据库，这样测试不会影响真实数据。用完记得 `clear()`或使用 `with`上下文管理器。
    
2. **`TestClient`的 `files`参数**：模拟 multipart 文件上传，每个元素是 `(field_name, (filename, content, content_type))`。
    
3. **`auth_headers`fixture**：统一管理认证头，避免每个测试重复写。
    
4. **测试隔离**：每个测试函数内的数据修改不会影响其他测试，因为 `client`fixture 每次新建一个 TestClient，但共享同一个 app 实例。如果测试之间需要完全隔离，可以在每个测试前重置应用状态（如清空数据库）。
    

## 总结

- **`TestClient`**​ 是 FastAPI 测试的核心，继承自 httpx，无需启动服务器，速度快。
    
- **依赖覆盖**：`app.dependency_overrides[original_dependency] = mock_function`是替换外部依赖的标准做法，适用于数据库、认证、第三方 API。
    
- **测试结构**：
    
    - 使用 pytest fixture 管理 `client`、认证头、测试数据。
        
    - 每个测试函数专注一个场景：成功、失败、边界条件、权限。
        
    - 断言状态码、响应体结构、错误消息。
        
    
- **文件上传测试**：通过 `files`参数构造 multipart 请求，可测试类型校验、大小限制等。
    
- **常见陷阱**：
    
    - 忘记清除 `dependency_overrides`导致后续测试受影响 → 用 `with app.dependency_overrides as overrides:`上下文管理器。
        
    - 测试间共享可变数据 → 每个测试前重新初始化测试数据库。
        
    - 异步测试需要 `pytest-asyncio`和 `async_client`（`TestClient`也有异步版本 `AsyncClient`）。
