### **第三阶段：Nginx 配置详解**

您的配置非常经典，实现了反向代理和SSL终结。我们来逐行分析 `/etc/nginx/sites-available/web`：

```nginx
server {
    # 监听443端口，并启用SSL协议
    listen 443 ssl;
    # 指定该配置块响应的域名，必须与您的樱花穿透域名完全一致
    server_name www.h01f0c017.nyat.app;

    # 指定SSL证书和私钥的绝对路径
    ssl_certificate /home/adorukw/file/ssl/certificate.crt;
    ssl_certificate_key /home/adorukw/file/ssl/private.key;

    #  location / 块匹配所有未匹配其他规则的请求
    location / {
        # 将接收到的所有请求转发到本地8080端口（即Apache）
        proxy_pass http://localhost:8080/;
        # 将原始请求的Host头部信息传递给后端服务器，这对WordPress正确识别域名至关重要
        proxy_set_header Host $http_host;
        # 以下两行是强烈建议添加的，用于传递客户端的真实IP和协议
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 匹配以 /fitness/ 开头的请求
    location /fitness/ {
        # 将这些请求转发到运行在7426端口的Node.js应用
        proxy_pass http://localhost:7426/;
        # 同样建议传递头部信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 匹配以 /words/ 开头的请求
    location /words/ {
        # 将这些请求转发到运行在7427端口的另一个Node.js应用
        proxy_pass http://localhost:7427/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**启用Nginx配置**
```bash
# 1. 创建从sites-available到sites-enabled的符号链接
sudo ln -s /etc/nginx/sites-available/web /etc/nginx/sites-enabled/

# 2. 删除默认的配置链接，避免冲突
sudo rm /etc/nginx/sites-enabled/default

# 3. 测试配置语法是否正确
sudo nginx -t
# 如果看到 "test is successful" 字样，说明配置正确

# 4. 重新加载Nginx以使新配置生效
sudo systemctl reload nginx
```

### 踩坑记录

1.  **樱花穿透隧道类型选择**
    *   **问题**：初期不了解隧道类型的区别。樱花穿透的 `HTTP/HTTPS` 隧道可以直接使用 80/443 端口，访问时无需在域名后加端口号，但**必须使用自己已备案的域名**。
    *   **解决方案**：由于没有备案域名，转而使用 `TCP` 隧道。但 `TCP` 隧道的访问方式是 **`域名:固定端口号`** (例如 `https://www.h01f0c017.nyat.app:26481`)，并且浏览器访问时必须使用 `HTTPS` 协议。

2.  **Nginx 监听端口配置**
    *   **问题**：因为外部通过 `HTTPS` 访问樱花穿透的特定端口，所以隧道指向的服务器内部端口也必须是 `443`，才能让 Nginx 进行标准的 SSL 终结。并且我开始尝试用nginx直接配置wordpress，如
```nginx
server {
    listen 80;
    server_name www.h01f0c017.nyat.app;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;
    server_name www.h01f0c017.nyat.app;
    ssl_certificate /home/adorukw/file/ssl/certificate.crt;
    ssl_certificate_key /home/adorukw/file/ssl/private.key;
    location / {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $http_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```
但弄不好
    *   **解决方案**：将 Nginx 的 `server` 块配置为 `listen 443 (这里绝对不能加ssl，我也不知道为什么);`，确保隧道流量能被正确接收和处理。

3.  **Nginx 反向代理配置的关键参数**
    *   **问题**：在配置 Nginx 反向代理到后端 WordPress (Apache) 时，如果仅配置 `proxy_pass` 而缺少必要的头部信息，会导致 WordPress 无法正确识别原始请求的域名，从而产生重定向错误或样式加载失败。
    *   **解决方案**：经过多次测试，发现 **`proxy_set_header Host $http_host;` 是必须且最关键的配置项**。此指令将客户端的原始 `Host` 头（即您的樱花穿透域名）完整地传递给后端的 Apache/WordPress，WordPress 据此生成正确的链接和重定向地址。

4.  **多余的配置项导致问题**
    *   **问题**：初期参考了一些网络教程，在 Nginx 配置中添加了 `proxy_set_header X-Forwarded-Proto $scheme;`，意图告诉后端这是 HTTPS 请求。然而，这与我特定的环境（TCP隧道 + 自定义端口）产生了冲突，反而导致了重定向循环或协议识别错误。
    *   **解决方案**：**果断移除了 `X-Forwarded-Proto` 等非必需配置项**，只保留最核心的 `proxy_pass` 和 `proxy_set_header Host $http_host;`，问题立刻得到解决。结论：配置并非越多越好，应根据实际环境进行精简和调试。

5.  **前端请求url**
    *   **问题**：因为后端用nginx反向代理多了一个'/example/'路径，因此原来前端的相对路径请求/api/就会因为少了一个/example/而失败
    *   **解决方案**：在前端加上一个BASE_URL即可，这个变量等于后端在nginx中配置的路径。

6.  **wordpress连接协议不一致**
    *   **问题**：因为前端请求中必须用https协议（樱花穿透强制规定）但是后端nginx和apache中都没有配置https，导致wordpress连接协议不一致。一些静态资源无法加载。

### 详细分类总结

#### 一、系统与环境初始化
-   `sudo apt update && sudo apt upgrade -y`：更新系统。
-   `sudo timedatectl set-timezone Asia/Shanghai`：设置服务器时区。

#### 二、Node.js 应用部署与管理 (PM2)
-   `sudo npm install -g pm2`：安装 PM2。
-   `pm2 start server.js --name fitness-reminder`：启动应用。
-   `pm2 startup`：生成 PM2 开机自启脚本。
-   `pm2 save`：保存当前进程列表，以便开机恢复。
-   `pm2 list`：查看当前由 PM2 管理的进程状态。

#### 三、Web 服务器与数据库环境搭建 (LEMP)
-   `sudo apt install nginx`：安装 Nginx。
-   `sudo apt install mariadb-server`：安装 MariaDB 数据库。
-   `sudo mysql_secure_installation`：进行数据库安全初始化。
-   `sudo apt install php-fpm php-mysql ...`：安装 PHP 及一系列扩展。

#### 四、WordPress 安装与配置
-   **数据库操作**：使用 `mysql -u root -p` 登录，为 WordPress 创建了数据库和用户。
-   **文件部署**：将 WordPress 文件复制到了 `/var/www/wordpress/` 目录。
-   **权限设置**：执行了 `sudo chown -R www-data:www-data /var/www/wordpress/` 等命令，正确设置了文件和目录权限。
-   **配置文件**：编辑了 `/var/www/wordpress/wp-config.php` 来连接数据库。

#### 五、Nginx 核心配置（主要工作）
-   你创建并编辑了主配置文件 `/etc/nginx/sites-available/web`。
-   **配置内容**：
    -   **反向代理**：将不同路径（如 `/fitness/`, `/words/`）代理到本地不同的 Node.js 应用端口（如 `7426`）。
